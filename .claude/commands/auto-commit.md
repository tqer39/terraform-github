# Auto Commit

変更ファイルを自動でステージングし、適切なコミットメッセージを生成してコミットする。

## 手順

### 1. 変更状態の確認

以下を並列で実行:

- `git status` で未追跡・変更ファイルを確認（`-u all` フラグは使用禁止）
- `git diff` でステージ済み・未ステージの変更内容を確認
- `git log --oneline -20` で直近のコミットスタイルを確認

### 2. 変更内容の分析

- 変更ファイルと差分を読み、変更の目的を把握する
- 直近のコミットメッセージのスタイル（言語、形式、スコープの使い方）を学習する

### 3. ステージング

- 変更ファイルを **個別に** `git add` する（`git add .` や `git add -A` は禁止）
- 以下のファイルは **絶対にステージしない**:
  - `.env`, `.env.*`
  - `credentials.json`, `*.secret`, `*.key`
  - シークレットや認証情報を含む可能性のあるファイル

### 4. Lint チェック

- `just lint` を実行する
- 失敗した場合:
  - エラー内容を確認し、`just fix` または手動で修正する
  - 修正したファイルを再ステージング（ステップ 3 のルールに従う）
  - `just lint` を再実行してパスを確認する
- **パスするまでコミットに進まない**

### 5. コミットメッセージの生成

```text
<emoji> <日本語の要約>

[optional body]
```

**絵文字プレフィックス:**

| 絵文字 | 用途 |
| -------- | ------ |
| 🚀 | 新機能（feat） |
| 🐛 | バグ修正（fix） |
| 📝 | ドキュメントのみの変更（docs） |
| 🎨 | コードの意味に影響しない変更（style） |
| ♻️ | リファクタリング（refactor） |
| ✅ | テストの追加・修正（test） |
| 🔧 | ビルドプロセスやツールの変更（chore） |

**ルール:**

- 要約は日本語で、変更内容を簡潔に説明する
- 複数の変更がある場合は body で箇条書きで補足する

### 6. コミット実行

- HEREDOC 形式でコミットメッセージを渡す:

```bash
git commit -m "$(cat <<'EOF'
<メッセージ>
EOF
)"
```

- `--no-verify` は **絶対に使わない**（pre-commit hook を尊重）
- コミット後に `git status` で成功を確認する

## 禁止事項

- `git add .` や `git add -A` の使用
- `--no-verify` フラグの使用
- `.env` やシークレットファイルのステージング
- `git push` の実行（コミットのみ）
- 変更がない場合の空コミット

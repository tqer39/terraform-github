# ドキュメントガイドライン

## ファイル構成

| 配置場所 | 言語 | 例 |
| ------- | ---- | -- |
| プロジェクトルート (`*.md`) | 英語 | README.md, CONTRIBUTING.md, CHANGELOG.md, CLAUDE.md |
| `./docs/*.ja.md` | 日本語 | README.ja.md, CONTRIBUTING.ja.md, architecture.ja.md |

## ベストプラクティス

- メインのプロジェクトドキュメントは英語を標準とする
- 日本語ドキュメントは `./docs/` 内の補完的な役割
- ファイル命名は明確で一貫性を保つ
- 更新時は対応する言語版も同期させる
- 必要に応じて `./docs/` 内にロケール別サブディレクトリを作成

## 開発日記

### 目的

- 主要な変更とその理由を記録
- 技術的な意思決定と実装の詳細を文書化
- 発生した問題と解決策を追跡
- 将来の開発者にコンテキストを提供

### 作成ルール

- 未コミットの変更や未文書化の作業がある場合に作成
- ファイル命名: `docs/dev-diary/YYYY-MM-DD.md`
- 当日の日付のファイルがなければ自動作成
- 前回の日記以降の全作業を記録

### フォーマット

各日記エントリには以下を含める:

1. 作業概要
2. 実装の詳細
3. テスト結果
4. 問題と解決策
5. 今後の検討事項
6. 開発者の気分と振り返り
7. リファクタリング候補

### タイミング

開発セッションの終了時、または依頼があった時に作成する。

**Tech Stack:** mise, pnpm, lefthook, cspell, GitHub Actions

- [ ] **Step 2: `mise.toml` の `[tools]` セクションに `node = "22"` を追加**

Expected: `v22.x.y` （`22.18.0` 以上の値）。

```bash

python3 -c "import sys; v=tuple(map(int, '$node_version'.split('.'))); sys.exit(0 if v >= (22,18,0) else 1)" && echo "OK: >=22.18.0" || echo "NG: <22.18.0"
```

Expected: `OK: >=22.18.0`

- [ ] **Step 5: `pnpm install --frozen-lockfile` が Node 22 で成功することを検証**

Run: `mise exec -- pnpm install --frozen-lockfile`

Expected: エラーなく完了。`node_modules/` が再生成される。

- [ ] **Step 6: lefthook pre-commit 全体が Node 22 で pass することを検証**

Run: `mise exec -- lefthook run pre-commit --all-files`

Expected:

- `cspell` フックが `Unsupported NodeJS version` エラーを出さずに pass する
- 全フックが pass（summary で 🥊 付きのフックがない）

注: この worktree の cspell は現在 v9.8.0（`package.json` 参照）。Node 22 でも v9 は動くので成功する想定。PR #1577 で cspell v10 にアップデートされたブランチでも Node 22 で pass することを後続のリモート検証で確認する。

## リスクと対処

- **mise で Node 22 を解決した CI とローカルでパッチバージョンに差異が出る可能性**: cspell v10 の要件 `>=22.18.0` を満たす限り問題ない。将来的に明示バージョン化したい場合は別タスクで扱う。
- **`pnpm/action-setup@v5` と mise の Node が競合する可能性**: pnpm action は `run_install: false` で Node を自前で用意しないため、PATH 上の mise shim が使われる想定。Step 6 の lefthook 検証で確認済み。
- **ローカル開発者が `mise install` を再実行しないと Node 22 を取得できない**: 既存 mise ワークフローに従う（README 追記などは不要）。
